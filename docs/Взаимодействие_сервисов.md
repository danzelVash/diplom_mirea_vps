# Взаимодействие сервисов

## 1. Назначение документа

Документ описывает базовое взаимодействие сервисов системы управления жилым помещением и фиксирует, какие сущности обрабатываются каждым сервисом. В архитектуре предусмотрены два контура:

- `Core` — центральные сервисы, работающие по сети при наличии интернета;
- `Edge` — локальный контур на `Raspberry Pi 5`, обеспечивающий сбор телеметрии и ограниченную offline-работу.

---

## 2. Основные сервисы

### 2.1. Core

- `api-gateway`
- `edge-bridge-service`
- `device-service`
- `context-service`
- `scenario-service`
- `voice-service`
- `vision-service`
- `notification-service`

### 2.2. Edge

- `edge-agent`

## 3. Роли сервисов

### `api-gateway`

Назначение:

- внешняя точка входа для клиента;
- приём пользовательских запросов;
- маршрутизация запросов в внутренние сервисы;
- предоставление агрегированных ответов.

Основные сущности:

- пользователь;
- команда пользователя;
- состояние системы;
- сценарий;
- инцидент.

### `edge-bridge-service`

Назначение:

- сетевой шлюз между `Core` и `edge-agent`;
- приём телеметрии с `Edge`;
- передача команд на `Edge`;
- организация синхронизации после восстановления связи.

Основные сущности:

- событие устройства;
- телеметрия;
- управляющая команда;
- статус связи;
- пакет синхронизации.

### `device-service`

Назначение:

- реестр датчиков и исполнительных устройств;
- хранение текущего состояния устройств;
- управление командами изменения состояния;
- контроль факта выполнения команды.

Основные сущности:

- датчик;
- устройство;
- команда;
- состояние устройства;
- комната;
- помещение.

### `context-service`

Назначение:

- агрегировать сенсорные данные, видео- и голосовые результаты;
- формировать актуальный контекст помещения;
- определять присутствие, занятость комнаты и режим среды.

Основные сущности:

- контекст помещения;
- контекст комнаты;
- контекст присутствия;
- состояние среды;
- объединённое событие.

### `scenario-service`

Назначение:

- хранить и исполнять правила автоматизации;
- принимать итоговое решение по управляющему воздействию;
- разрешать конфликтующие сценарии;
- запускать штатные и защитные сценарии.

Основные сущности:

- сценарий;
- правило;
- условие;
- действие;
- приоритет;
- решение.

### `voice-service`

Назначение:

- обработка голосовой команды;
- преобразование аудио в текст;
- определение намерения пользователя;
- формирование структурированной команды.

Основные сущности:

- аудиофрагмент;
- распознанный текст;
- намерение;
- голосовая команда;
- целевая сущность.

### `vision-service`

Назначение:

- анализ кадра или потока;
- идентификация пользователя;
- фиксация факта появления неизвестного лица;
- генерация событий безопасности.

Основные сущности:

- видеокадр;
- человек;
- лицо;
- результат идентификации;
- событие безопасности.

### `notification-service`

Назначение:

- отправка уведомлений;
- регистрация тревожных событий;
- журналирование инцидентов;
- эскалация критических ситуаций.

Основные сущности:

- уведомление;
- тревога;
- инцидент;
- канал доставки;
- статус отправки.

### `edge-agent`

Назначение:

- локальный агент на `Raspberry Pi 5`;
- взаимодействие с физическими устройствами и локальными шлюзами;
- буферизация событий;
- исполнение команд;
- выполнение ограниченного offline-набора правил.

Основные сущности:

- локальное событие;
- локальное состояние;
- очередь синхронизации;
- локальный сценарий;
- статус подключения.

## 4. Базовая схема взаимодействия

```text
Пользователь / клиент
        |
        v
   api-gateway
        |
        +---------------------> voice-service
        |
        +---------------------> vision-service
        |
        +---------------------> device-service
        |
        +---------------------> context-service
        |
        +---------------------> scenario-service
        |
        +---------------------> notification-service
        |
        v
 edge-bridge-service <------------------> edge-agent (Raspberry Pi 5)
                                           |
                                           v
                                 датчики / Zigbee / камеры / устройства
```

---

## 5. Основные потоки данных

## 5.1. Поток телеметрии

1. `edge-agent` получает событие от датчика или устройства.
2. Событие отправляется в `edge-bridge-service`.
3. `edge-bridge-service` передаёт данные в `device-service`.
4. `device-service` обновляет текущее состояние.
5. Обновление передаётся в `context-service`.
6. `context-service` пересчитывает контекст помещения.
7. При необходимости `scenario-service` принимает решение.
8. Итоговая команда возвращается через `edge-bridge-service` в `edge-agent`.

## 5.2. Поток голосовой команды

1. Пользователь произносит команду.
2. Аудио может поступить через клиентский интерфейс или через локальный контур `Edge`.
3. `voice-service` преобразует аудио в структурированную команду.
4. Результат передаётся в `context-service` и `scenario-service`.
5. Если команда прямая, `scenario-service` формирует управляющее действие.
6. Команда поступает в `device-service`.
7. Затем команда доставляется на `edge-agent` через `edge-bridge-service`.

## 5.3. Поток видеоаналитики

1. Кадр или поток поступает с камеры на `Edge`.
2. `edge-agent` отправляет кадр или ссылку на него в `edge-bridge-service`.
3. `vision-service` выполняет анализ.
4. Результат идентификации поступает в `context-service`.
5. Если обнаружен посторонний, событие передаётся в `notification-service`.
6. При необходимости `scenario-service` инициирует защитный сценарий.

## 5.4. Поток сценарного управления

1. `context-service` формирует новый контекст.
2. `scenario-service` проверяет условия сценариев.
3. Выбирается допустимое действие с учётом приоритета.
4. Решение отправляется в `device-service`.
5. `device-service` подготавливает команду для устройства.
6. Команда передаётся на `Edge` через `edge-bridge-service`.
7. `edge-agent` исполняет команду и возвращает результат выполнения.

---

## 6. Распределение ответственности между Core и Edge

## 6.1. Что выполняется в Core

- полная агрегация и анализ событий;
- обработка сложных сценариев;
- голосовая аналитика;
- видеоаналитика;
- централизованное хранение состояния;
- генерация уведомлений и инцидентов.

## 6.2. Что выполняется на Edge

- приём событий от физических устройств;
- локальное исполнение команд;
- поддержание локального снимка текущего состояния;
- накопление событий при потере связи;
- выполнение ограниченного набора правил offline.

---

## 7. Offline-модель для Raspberry Pi 5

### 7.1. Общий принцип

`edge-agent` должен иметь урезанный, но рабочий локальный контур управления. Это означает, что при потере связи система не должна полностью останавливаться.

### 7.2. Что должно работать offline

- локальный сбор телеметрии;
- хранение последнего состояния устройств;
- выполнение базовых реакций по простым правилам;
- выполнение ограниченного набора критичных сценариев;
- локальное включение и выключение устройств;
- накопление событий для последующей синхронизации.

### 7.3. Что может быть ограничено offline

- сложная межкомнатная аналитика;
- полнофункциональная видеоаналитика;
- тяжёлое распознавание голоса;
- расширенное разрешение конфликтов сценариев;
- удалённые уведомления, зависящие от внешней сети.

### 7.4. Какие сценарии уместно оставить на Edge

- включение света по движению;
- базовые реакции на открытие двери;
- локальное управление климатом по пороговым значениям;
- простые защитные реакции;
- исполнение ранее синхронизированных сценариев с простыми условиями.

---

## 8. Организация сетевого взаимодействия

В нормальном режиме при наличии интернета взаимодействие происходит по сети между `edge-agent` и центральными сервисами.

Базовая модель:

- клиент работает с `api-gateway`;
- внутренние сервисы `Core` обмениваются данными по внутренней сети;
- `edge-agent` связан с `edge-bridge-service` по отдельному защищённому каналу;
- все управляющие команды к физическим устройствам проходят через `edge-agent`.

Такое разделение позволяет:

- вынести тяжёлую вычислительную часть с `Raspberry Pi 5`;
- сохранить локальную работоспособность при потере связи;
- централизовать принятие решений на сервере;
- не давать внешним клиентам прямой доступ к физической плате.
